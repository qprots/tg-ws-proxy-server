# TG WS Proxy Серверная часть

**Локальный SOCKS5-прокси** для Telegram любой версии, который **улучшает связность с Telegram**, перенаправляя трафик через WebSocket-соединения. Данные передаются в оригинальном зашифрованном виде, сторонние серверы не требуются.

## Техническое описание

Это **серверная реализация** моста-прокси, изначально созданная для собственных нужд по стабилизации работы Telegram на удаленных серверах. За основу был взят и переработан проект [tg-ws-proxy](https://github.com/Flowseal/tg-ws-proxy) от [Flowseal](https://github.com/Flowseal).

Модифицируйте под ваши задачи.

### Технологический стек
*   **Ядро**: Python 3.12 (асинхронная реализация на `asyncio`).
*   **Безопасность**: Библиотека `cryptography` для обнаружения и обработки MTProto-пакетов.
*   **Проксирование**: [redsocks](https://github.com/darkk/redsocks) для перехвата и пересылки TCP-трафика на уровне системы.
*   **Сеть**: `iptables` для прозрачного перехвата трафика подсетей Telegram.
*   **Развертывание**: Docker Compose (multi-stage сборка для минимизации размера образа) + Systemd (для управления правилами сети).

### Принцип работы
1.  **Перехват трафика**: Скрипт `iptables-tg.sh` настраивает ядро Linux на пересылку пакетов, идущих к IP-адресам Telegram, в локальный порт `redsocks` (`12345`).
2.  **SOCKS5-адаптация**: `redsocks` принимает TCP-соединение и передает его в основной прокси-сервис по протоколу SOCKS5 (порт `10080`).
3.  **WebSocket-бриджинг**:
    *   Прокси анализирует первый пакет соединения (init-пакет MTProto).
    *   Определяет, к какому дата-центру (DC) идет обращение.
    *   Создает TLS/WebSocket соединение с официальными доменами Telegram (например, `wss://kws2.web.telegram.org/apiws`).
    *   Инкапсулирует MTProto-трафик в WebSocket-фреймы, обходя тем самым ограничения на прямой TCP-трафик для определенных диапазонов IP.
4.  **Синхронизация**: Если WebSocket соединение не удается установить (например, при 302 редиректе), система может откатиться на прямой TCP-проброс (если доступно).

## Быстрый старт (Docker Compose)

```bash
docker compose up -d
```

Прокси запустится на порту `10080` с автоматическим перезапуском.

## Глобальное прозрачное проксирование

Для перенаправления **всего** TCP-трафика к подсетям Telegram через прокси (для любого приложения на машине):

```bash
# Запустить прокси + redsocks
docker compose up -d

# Включить перенаправление через iptables
sudo ./iptables-tg.sh start

# Проверить правила
sudo ./iptables-tg.sh status

# Выключить перенаправление
sudo ./iptables-tg.sh stop
```

**Архитектура:**

```
Любое приложение -> TCP на IP Telegram
    -> iptables REDIRECT -> redsocks (:12345)
    -> SOCKS5 -> tg-ws-proxy (:10080)
    -> WSS -> Telegram DC
```

|---|---|

| `91.108.4.0/22` | Telegram |
| `91.108.8.0/22` | Telegram |

## Скрипты автоматизации

Для удобства развертывания на новые ноды предусмотрены скрипты. **Перед использованием их необходимо подправить под свои нужды** (пути, порты, параметры SSH):

| Скрипт | Описание |
|---|---|
| `./deploy.sh <host>` | Деплой в одну кнопку на удаленный сервер |
| `./test_connectivity.sh <host>` | Автоматическая проверка связности и конфигурации |
